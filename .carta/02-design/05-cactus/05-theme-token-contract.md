# Theme Token Contract

cactus is a domain-agnostic canvas engine. It renders chrome, nodes, edges, and
grid — all of which need color. But cactus must not *decide* what those colors
are: that is a host concern. A Luminous canvas, a different app embedding
cactus, and a Storybook story all want the same engine wearing different paint.

This doc defines the boundary: cactus declares a fixed set of **named color
slots** — the contract — and ships two **themes** that fill them. Consumers
either use a shipped theme or fill the slots themselves.

## The Boundary Problem

Before this contract, `cactus.css` read `--fg`, `--overlay`, `--accent`,
`--danger`, and friends directly. Those names are **Luminous's** domain
vocabulary, defined in `client-next/src/index.css`. cactus reached *up* into its
consumer's namespace — it only rendered correctly by the coincidence that
Luminous happened to define those exact names.

This violates the engine/domain split (see doc02.01, D8). Visual concerns belong
in cactus, but a *theme* is not a visual concern — it is a host decision about
brand and appearance. The fix is a contract: cactus owns the **slot names**, the
consumer owns the **values**.

## Three Tiers

```
TIER 1  cactus declares      var(--cactus-fg, #111827)        ← in cactus.css, every usage
TIER 2  a theme fills        .cactus-theme-dark { --cactus-fg: #e8e0d8 }
TIER 3  a consumer maps      [data-theme] { --cactus-fg: var(--fg) }   ← optional
```

- **Tier 1 — the contract.** Every color in cactus source is `var(--cactus-NAME, FALLBACK)`. The fallback is mandatory: with zero theme CSS loaded, cactus still renders a usable light appearance. The set of names is closed — adding one is a contract change.
- **Tier 2 — a theme.** A CSS rule that assigns a value to *every* token in the contract. cactus ships two (`cactus-theme-light`, `cactus-theme-dark`). Applied via a class on any ancestor of the canvas.
- **Tier 3 — a consumer mapping.** A host with its own theme system (like Luminous) skips the shipped themes and instead maps its domain tokens onto the contract. Pure pass-through, no new values.

A consumer uses exactly one of Tier 2 *or* Tier 3 — never both.

## Token Inventory

Nineteen tokens. Seventeen colors, two shadows. Every one has a fallback. The
canonical name list lives in `cactus-tokens.ts` (`CACTUS_TOKENS`).

Notes from the migration:

- **`--cactus-canvas-bg` is new.** Pre-contract, `Canvas.tsx` set no background and `DotGrid` defaulted to `transparent` — the backdrop leaked in from the host's `<body>`. cactus now owns its own backdrop.
- **`--bg-surface` was a stray.** It appeared once (edge-label halo stroke) as a typo-twin of `--surface`, never defined, always falling back to `white`. It folded into `--cactus-canvas-bg` — the color the label actually sits on.
- **`NodeShell` consumed theme colors via Tailwind utilities** (`bg-surface`, `outline-accent-subtle`, `border-border-subtle`, `bg-fg-subtle`…). Those utilities are generated by *Luminous's* `@theme inline` block — a second, less visible boundary leak that a `var()` grep misses. They were converted to inline `var(--cactus-*)` styles; layout-only utilities (`flex`, `rounded-lg`) stay. This added `--cactus-fg-subtle` and `--cactus-border-subtle`.
- **`accent` is two tokens, not one.** Luminous distinguishes `--accent-subtle` (a brighter solid accent, used for selection outlines and the connection-handle dot) from `--accent-10` (a translucent wash for selection-rect fill and menu highlight). The contract keeps both: `--cactus-accent-subtle` is the solid variant, `--cactus-selection` the wash.
- **No dedicated edge color.** Edges stroke with `--cactus-fg-muted`. If edges ever need to theme independently of muted text, that is a future `--cactus-edge` — a contract addition, not a silent change.

## One Contract, Two Themes

The question this contract has to answer: cactus ships a light theme *and* a dark
theme, but there must be exactly one contract. How?

**The contract is the set of token names. A theme is an assignment over that
exact set.** Light and dark are not two contracts — they are two
*implementations of the same interface*. The invariant:

> Every shipped theme assigns a value to **every** token in the inventory, and
> assigns values to **no other** `--cactus-*` token.

This is what makes the themes interchangeable: swapping the class on the
ancestor element can never leave a token unfilled or introduce one cactus
doesn't read. The fallbacks in Tier 1 are, in effect, a third implicit theme
(light) — which is why the light fallbacks and `cactus-theme-light` agree
value-for-value.

```css
/* cactus-themes.css — both themes, one file, same token set */
.cactus-theme-light {
  --cactus-canvas-bg: #ffffff;
  --cactus-fg: #111827;
  /* ... all 19 ... */
}
.cactus-theme-dark {
  --cactus-canvas-bg: #1e1a17;
  --cactus-fg: #e8e0d8;
  /* ... all 19 ... */
}
```

The token name list lives once, as data, in `cactus-tokens.ts` (or a comment
block both themes are checked against). A unit test asserts both shipped themes
define exactly that set — no drift, no missing slot. That test *is* the
enforcement of "one contract."

### Applying a theme

```html
<div class="cactus-theme-dark">
  <Canvas> … </Canvas>
</div>
```

Themes cascade through CSS custom properties, so the class can sit on any
ancestor — the canvas root, a panel, or `<html>`.

## Bring Your Own Theme

A host with its own theme system (Luminous has light/dusk/ground, switched via
`data-theme`) does not use the shipped themes. It maps its domain tokens onto
the contract once:

```css
/* client-next/src/index.css — Luminous fills the cactus contract */
:root {
  --cactus-canvas-bg:     var(--canvas);
  --cactus-fg:            var(--fg);
  --cactus-fg-muted:      var(--fg-muted);
  --cactus-fg-subtle:     var(--fg-subtle);
  --cactus-overlay:       var(--overlay);
  --cactus-surface:       var(--surface);
  --cactus-surface-alt:   var(--surface-alt);
  --cactus-border:        var(--border);
  --cactus-border-subtle: var(--border-subtle);
  --cactus-accent:        var(--accent);
  --cactus-accent-subtle: var(--accent-subtle);
  --cactus-selection:     var(--accent-10);
  --cactus-on-accent:     var(--on-accent);
  --cactus-danger:        var(--danger, #dc2626);
  --cactus-danger-subtle: var(--danger-10, #fef2f2);
  --cactus-grid-dot:      var(--dot-grid);
  --cactus-resize-handle: var(--resize-handle);
  --cactus-shadow-sm:     var(--shadow-sm);
  --cactus-shadow-lg:     var(--shadow-lg);
}
```

Because the mapping is at `:root` and the right-hand side reads Luminous's
own per-`data-theme` vars, all three Luminous themes flow through the contract
automatically — no per-theme cactus block needed. The host gets N themes from
one mapping; cactus stays unaware of how many themes exist.

## Rules

1. **No raw color in cactus source.** Every color is `var(--cactus-NAME, FALLBACK)`. Fallback is mandatory.
2. **The contract is closed.** Adding, removing, or renaming a `--cactus-*` token is a contract change — update this doc, the inventory, and both shipped themes together.
3. **cactus never reads a non-`--cactus-*` var.** Reading `--fg` or any host token is the boundary violation this contract exists to prevent.
4. **A shipped theme fills the whole set, exactly.** Enforced by test.
5. **Node *content* is out of scope.** The contract covers cactus chrome — toolbars, menus, the `NodeShell`/`NodeContainer` frame, edges, grid, selection. The colors *inside* a node body come from the domain pack's renderer and use domain tokens (e.g. `--color-kind-*`). The boundary runs through the node: frame is cactus, fill is domain.
